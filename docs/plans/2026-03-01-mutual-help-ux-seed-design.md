# 互相帮助 MVP 体验优化与演示数据设计

## 目标
- 一键生成可演示的完整数据链路（求助、帮助、通知），方便演示与验收。
- 关键页面补齐加载态、空状态与操作反馈，提升产品级体验。

## 方案选择
- **方案 A（最轻量）**：仅新增演示数据入口 + 少量空状态。优点最快落地；缺点体验提升有限。
- **方案 B（采用）**：演示数据入口 + 关键页面加载/空状态 + 表单与操作反馈。体验与投入平衡。
- **方案 C（更完整）**：方案 B + 通知详情增强 + 更多演示场景。改动更大。

## 范围与入口
### 演示数据入口
- 新增 `/demo` 页面（主布局下），仅登录可见。
- 仅 **开发环境** 显示导航入口，并限制 API 访问，避免线上误触。
- 页面提供「生成/重置演示数据」按钮与演示账号说明。

### 演示数据规则
- 演示用户统一使用 `demo_` 前缀（如 `demo_owner`）。
- 演示求助标题统一使用 `【演示】` 前缀，便于清理。
- 生成前先清理同前缀演示数据，不影响真实数据。
- 若城市/小区为空，自动补齐一组默认种子。

### 数据构成（固定示例）
- 演示账号：`demo_owner` / `demo_helper1` / `demo_helper2`，默认密码 `123456`。
- 当前登录用户：
  - **OPEN** 求助（含 2 个演示帮助申请）
  - **COMPLETED** 求助（已选中并完成）
- 演示发布者（demo_owner）：
  - **OPEN** 求助（广场展示）
  - **MATCHED** 求助（当前用户已被选中，展示“我的帮助”）

## 交互体验优化
- 为主路由段添加统一 `loading.tsx`，覆盖：
  - 帮助广场、求助详情、发布求助、我的求助、我的帮助、通知、演示数据页
- 空状态增加明确引导：
  - 我的求助 → 引导发布
  - 我的帮助 → 引导去广场
  - 通知 → 引导查看广场
- 表单/操作补齐禁用态与“处理中”文案（已有的补缺失处）。

## 错误处理
- 演示数据按钮返回可读错误提示（权限/环境/网络）。
- 前端继续沿用统一 `{ error }` 提示模式。

## 测试策略
- 新增服务层测试：演示数据可重复执行，生成数量稳定且不报错。
- 手动冒烟：跑 `npm run dev`，验证页面加载/空态/演示数据生效。
